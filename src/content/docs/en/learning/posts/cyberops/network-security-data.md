## MODULE OVERVIEW

Network Security Monitoring (NSM) involves the collection and analysis of diverse data types to identify, investigate, and mitigate security incidents. This module defines the hierarchy of network data—ranging from high-level statistical summaries to granular full packet captures—and examines the infrastructure tools required to process this telemetry. Effective cybersecurity operations require an integrated approach to log management, utilizing Security Information and Event Management (SIEM) and Security Orchestration, Automation, and Response (SOAR) platforms to unify disparate data sources into a cohesive defensive posture.

## SECTION 1 — NETWORK MONITORING DATA CATEGORIES

There are six primary categories of network monitoring data. Understanding the distinctions between them is critical for exam success.

### 1.1 Alert Data

Alert data consists of messages generated by **Intrusion Prevention Systems (IPS)** or **Intrusion Detection Systems (IDS)** in response to traffic that violates a rule or matches the signature of a known exploit. A network IDS (NIDS) such as **Snort** comes pre-configured with rules for known exploits.

**Exam tip:** Alert data is generated *by IDS/IPS devices*. It is the output of automated rule or signature matching, not a raw capture.

**NIDS Validation — Snort Example:**

To verify that a Snort NIDS is operational, analysts can visit `testmyids.com`. This site returns a page displaying `uid=0(root) gid=0(root) groups=0(root)`, which matches a Snort signature and triggers a test alert.

| Snort Rule Field | Value |
| :--- | :--- |
| Rule SID | `2100498` |
| Signature Pattern | `uid=0(root)` |
| Alert Message | `GPL ATTACK_RESPONSE id check returned root` |
| Detection Trigger | Any IP receiving content matching `uid=0\|28\|root\|29\|` from an external source |

Alerts are rendered readable and searchable by tools such as **Sguil** and **Squert**, both part of the Security Onion NSM suite.

---

### 1.2 Session Data

Session data is a **record of a conversation between two network endpoints** (typically a client and a server). It contains metadata *about* the session, not the actual content transmitted.

Session data is identified by the **5-tuple**:

| 5-Tuple Field | Description |
| :--- | :--- |
| Source IP Address | IP of the originating host |
| Destination IP Address | IP of the responding host |
| Source Port | Protocol port of the originating host |
| Destination Port | Protocol port of the responding host |
| Protocol | Layer 3 protocol type (e.g., TCP, UDP) |

Additional session metadata typically includes: session ID, amount of data transferred by each endpoint, and session duration.

**Exam tip:** Session data describes *the flow*, not the content. It answers "who talked to whom, for how long, and how much data was exchanged."

**Zeek (formerly Bro)** is a key NSM tool that generates session logs. A Zeek connection log record includes the following fields:

| Field | Meaning |
| :--- | :--- |
| `ts` | Session start timestamp |
| `uid` | Unique session ID |
| `id.orig_h` | Originating (source) IP address |
| `id.orig_p` | Originating (source) port |
| `id.resp_h` | Responding (destination) IP address |
| `id.resp_p` | Responding (destination) port |
| `proto` | Layer 4 protocol |
| `duration` | Duration of the session |
| `orig_bytes` | Bytes sent by originating host |
| `resp_bytes` | Bytes sent by responding host |
| `orig_pkts` | Packets from originating host |
| `resp_pkts` | Packets from responding host |

---

### 1.3 Transaction Data

Transaction data consists of the **specific messages exchanged during a network session** — the requests and responses themselves. While a session represents the entire conversation, a transaction is a single request/response pair within it.

**Example:** An HTTP session may involve multiple transactions: an HTTP GET request and the server's HTTP response. These are logged in a web server access log or by a NIDS like Zeek.

**Key distinction:**
- **Session** = all traffic involved in the overall conversation
- **Transaction** = the specific request or reply within that conversation

Transaction data includes device-specific server and host logs (e.g., Apache access logs, IIS logs, email server logs).

---

### 1.4 Full Packet Capture (FPC)

Full Packet Capture is the **most comprehensive and storage-intensive** type of NSM data. Unlike session data, FPC records every bit of traffic, including the actual **payload and content** of communications.

FPC enables reconstruction of entire conversations, including:
- Text of email messages
- HTML content of web pages
- Files transferred across the network (**extracted content**)
- Payload analysis for malware detection

**Primary tool:** **Wireshark** (GUI) and **tcpdump** (command-line) are the standard tools for generating and viewing full packet captures.

**Exam tip:** FPC contains detailed **protocol and payload information** for all traffic. It is the only data type that captures the actual file content flowing through the network. Extracted content (e.g., email attachments, downloaded files) is recoverable from FPCs.

---

### 1.5 Statistical Data

Statistical data is **derived from the analysis of other network data forms**. It does not capture raw traffic but instead produces summaries, baselines, and anomaly indicators through advanced analytics.

Statistical data is used to:
- Establish baselines of normal network behavior
- Detect anomalies by comparing current traffic to baselines
- **Describe or predict network behavior** through machine learning and predictive analytics

Techniques include **Network Behavior Analysis (NBA)** and **Network Behavior Anomaly Detection (NBAD)**, which analyze NetFlow or IPFIX telemetry data.

**Example tool:** **Cisco Cognitive Threat Analytics (Cognitive Intelligence)** — a cloud-based product that uses machine learning and statistical modeling to find malicious activity that has bypassed security controls or entered through unmonitored channels, and is operating inside an enterprise network.

**Exam tip:** Statistical data *summarizes or analyzes* flow/performance data. It is the answer to questions about describing or predicting network behavior.

---

### 1.6 Extracted Content

Extracted content refers to **files and objects recovered from full packet captures** — for example, files attached to emails or files downloaded from the internet. Extracted content is analyzed for malware or policy violations.

---

### Summary Comparison Table

| Data Type | What It Contains | Primary Use Case | Storage Cost |
| :--- | :--- | :--- | :--- |
| **Alert Data** | IDS/IPS rule match notifications | Immediate threat notification | Low |
| **Session Data** | 5-tuple + flow metadata | Flow accounting, timeline reconstruction | Low |
| **Transaction Data** | Request/response messages, server logs | Application-layer analysis, DLP | Medium |
| **Full Packet Capture** | Complete payload + protocol data | Deep forensic analysis, content extraction | Very High |
| **Statistical Data** | Derived analysis, baselines, anomaly scores | Behavioral analysis, anomaly detection | Low |
| **Extracted Content** | Files from emails, downloads from FPC | Malware analysis, policy enforcement | Medium |

## SECTION 2 — HOST AND SERVER LOG ANALYSIS

### 2.1 Windows Event Log Classification

Windows systems categorize security and operational events into specific types to facilitate troubleshooting and auditing. These are recorded in the **Windows Event Viewer**.

| Event Type | Description | Operational Example |
| :--- | :--- | :--- |
| **Error** | Significant problem indicating data loss or functional failure | Service failing to load during startup |
| **Warning** | Potential future problem; non-critical, system recovered | Low disk space notification |
| **Information** | Describes the **successful operation** of an application, driver, or service | Successful loading of a network driver |
| **Success Audit** | Records a **successful** security access attempt | A user successfully logging on to the system |
| **Failure Audit** | Records a **failed** security access attempt | An unauthorized user failing to log in remotely; unauthorized access attempt to a network drive |

**Exam tip — Quick mapping:**
- Disk error during backup → **Error**
- Remote login unsuccessfully attempted by unauthorized user → **Failure Audit**
- Disk space getting low → **Warning**
- Successful application/driver/service operation → **Information**

### 2.2 Windows Log Types

Windows maintains several distinct logs. For the exam, the key distinction is:

| Log Type | Records |
| :--- | :--- |
| **Security Logs** | Login attempts, file/object access audits (Success Audit and Failure Audit events) |
| **Application Logs** | Events from specific software applications |
| **System Logs** | Events from OS components and drivers |
| **Setup Logs** | Events related to application installation |

**Exam tip:** Security logs record **login attempts and operations related to file or object access**.

---

### 2.3 Syslog Severity Levels

The standard Syslog protocol classifies messages using a severity scale from 0 (most critical) to 7 (least critical).

| Value | Severity | Technical Meaning |
| :--- | :--- | :--- |
| 0 | Emergency | System is unusable |
| 1 | Alert | Action must be taken immediately |
| 2 | Critical | Critical conditions; system failure indicated |
| 3 | Error | Non-urgent failure; resolve within a specific timeframe |
| 4 | Warning | Error does not exist yet but will occur if condition is unaddressed |
| 5 | Notice | Unusual event; does not require immediate action |
| 6 | Informational | Messages regarding normal system operations |
| 7 | Debug | Detailed messages for development and troubleshooting |

**Syslog Priority (PRI) Formula:**

$$\text{Priority} = (\text{Facility} \times 8) + \text{Severity}$$

The Priority value appears as the first value in a syslog packet, enclosed in angle brackets (e.g., `<34>`). Facility codes 15–23 (local0–local7) have no fixed keyword and can be assigned contextually.

---

### 2.4 Server Logs

Network application servers maintain access and error logs that are primary sources of NSM data.

**Apache Web Server Access Log Example:**
```
203.0.113.127 - dsmith [10/Oct/2016:10:26:57 -0500] "GET /logo_sm.gif HTTP/1.0" 200 2254
```

**Microsoft IIS Access Log Example:**
```
6/14/2016, 16:22:43, 203.0.113.24, -, W3SVC2, WEB3, 198.51.100.10, 80, GET, /home.htm, -, 200, 6, 15321
```

**DNS Proxy Logs** are especially important — they document all DNS queries and responses, and are critical for identifying hosts that visited malicious sites, DNS exfiltration, and connections to C2 (command-and-control) servers.

## SECTION 3 — TELEMETRY TOOLS AND PLATFORMS

### 3.1 tcpdump

**tcpdump** is a command-line packet analyzer that can display packet captures in real time or write them to a file. It captures detailed packet protocol and content data (Full Packet Capture).

**Wireshark** is a GUI-based packet analyzer built on tcpdump functionality.

**Exam tip:** tcpdump = command-line tool for full packet capture. It is NOT used to analyze NetFlow data or statistical logs.

---

### 3.2 NetFlow

**NetFlow** is a protocol developed by Cisco for network troubleshooting and session-based accounting. It collects **metadata about packet flows**, not the actual content of packets.

NetFlow was the basis for the **IPFIX** (Internet Protocol Flow Information Export) IETF standard. IPFIX is based on Cisco NetFlow Version 9.

#### NetFlow 5-Tuple (mandatory fields in all flow records):

| Field | Description |
| :--- | :--- |
| Source IP Address | IP of the traffic originator |
| Destination IP Address | IP of the traffic destination |
| Source Port | Originating port number |
| Destination Port | Destination port number |
| Protocol | Layer 3 protocol type |

#### All NetFlow Flow Records Also Contain:

| Field | Description |
| :--- | :--- |
| **Beginning timestamp** | Start time of the flow |
| **Ending timestamp** | End time of the flow |
| Total packets | Packet count for the flow |
| Total bytes | Byte count for the flow |

**Exam tip — NetFlow key facts:**
- NetFlow does **NOT** capture the actual content (payload) of packets
- All flow records contain the 5-tuple + **beginning timestamp** + **ending timestamp**
- NetFlow data is viewed with tools like **nfdump** (command-line) or FlowViewer (GUI)
- NetFlow **cannot** be viewed with tcpdump
- NetFlow provides metadata (flow records), **not** full packet captures

#### NetFlow Flow Architecture:

```
Network Device (Exporter) → NetFlow Collector → nfdump / FlowViewer
```

---

### 3.3 Application Visibility and Control (AVC)

**Cisco AVC** combines multiple technologies to recognize, analyze, and control over **1,000 applications**, including VoIP, video, email, file sharing, gaming, P2P, and cloud-based applications.

#### AVC Architecture Modules:

| Module | Function |
| :--- | :--- |
| **Application Recognition** | Identifies applications using **NBAR2** (L3–L7 data) |
| **Metrics Collection** | Collects bandwidth, response time, latency, packet loss, jitter |
| **Management and Reporting** | Exports data to management tools; generates reports |
| **Management and Control** | Provisions the network; enforces application policies and QoS |

**NBAR2** (Next-Generation Network-Based Application Recognition) is the Cisco engine deployed in the **Application Recognition** module of AVC. It identifies applications by their **signatures** (not port numbers), enabling granular visibility into user behavior (e.g., teleconferencing vs. P2P file sharing).

**Exam tip:** AVC uses **NBAR2** to discover and classify applications responsible for network traffic. NBAR2 is deployed in the **Application Recognition** module.

---

### 3.4 Content Filter Logs

Devices that perform content filtering create logs that are valuable for NSM. Two key Cisco appliances:

| Device | Primary Function | Log Types |
| :--- | :--- | :--- |
| **Cisco Email Security Appliance (ESA)** | Filters and monitors email traffic | 30+ logs: antivirus, antispam, block/allow list decisions, delivery logs |
| **Cisco Web Security Appliance (WSA)** | Acts as a web proxy; filters HTTP traffic | ACL decision logs, malware scan logs, web reputation filtering logs, inbound/outbound transaction logs |

**Exam tip:** Both the **Web Security Appliance (WSA)** and **Email Security Appliance (ESA)** create logs of suspicious content detected in application traffic.

---

### 3.5 Proxy Logs

**Proxy servers** act as intermediaries for network clients. They log all requests and responses, enabling detection of malicious traffic and supporting **Data Loss Prevention (DLP)**.

#### Squid Web Proxy Log (Squid-native format):

```
1265939281.764 19478 172.16.167.228 TCP_MISS/200 864 GET http://www.example.com/images/home.png - NONE/- image/png
```

| Field | Example Value | Meaning |
| :--- | :--- | :--- |
| Timestamp | `1265939281.764` | Unix epoch timestamp with milliseconds |
| Elapsed | `19478` | Milliseconds taken for the transaction |
| Client IP | `172.16.167.228` | IP address of the requesting client |
| Result Code | `TCP_MISS/200` | Cache result / HTTP response code |
| Size | `864` | Bytes of data delivered |
| Method | `GET` | HTTP request method |
| URI | `http://www.example.com/images/home.png` | Resource URL requested |
| Peer Code | `NONE/-` | Neighbor cache consulted |
| Content Type | `image/png` | MIME type from HTTP response header |

Popular web proxy software: **Squid**, CCProxy, Apache Traffic Server, WinGate.

#### Cisco Umbrella (DNS Proxy):

**Cisco Umbrella** (formerly OpenDNS) is a hosted DNS security service that monitors DNS queries to identify connections to C2 servers or data exfiltration attempts. It applies real-time threat intelligence to DNS access management.

DNS Proxy Log Example:
```
"2015-01-16 17:48:41", "ActiveDirectoryUserName", "ActiveDirectoryUserName, ADSite, Network",
"10.10.1.100", "24.123.132.133", "Allowed", "1 (A)", "NOERROR", "domain-visited.com.",
"Chat, Photo Sharing, Social Networking, Allow List"
```

| Field | Meaning |
| :--- | :--- |
| Timestamp | UTC time of the DNS request |
| Identities | All identities (user, AD site, network) associated with the request |
| Internal IP | Client IP making the request |
| External IP | External IP that made the request |
| Action | Whether the request was Allowed or Blocked |
| Query Type | Type of DNS record requested (e.g., A, AAAA, MX) |
| Response Code | DNS response (e.g., NOERROR, NXDOMAIN) |
| Domain | Domain name requested |
| Categories | Content categories and policy decisions |

## SECTION 4 — NGFW EVENTS

**Next-Generation Firewalls (NGFW)** extend security beyond Layer 4 (IP/port) to the application layer. Cisco NGFW devices use **Firepower Services**, which include:
- Application Visibility and Control (AVC)
- Next-Generation IPS (NGIPS)
- URL filtering (reputation and category-based)
- Advanced Malware Protection (AMP)

### NGFW Event Types

| Event Type | Trigger | Key Details Logged |
| :--- | :--- | :--- |
| **Connection Event** | Sessions detected directly by the NGIPS | Timestamps, source/destination IPs, access control rule matched |
| **Intrusion Event** | Malicious activity detected that affects availability, integrity, or confidentiality | Date, time, exploit type, source/target context |
| **Host or Endpoint Event** | A host first appears on the network | Device hardware, IP addressing, last known presence |
| **Network Discovery Event** | Changes detected in monitored network (hosts, applications) | Changes to known hosts or applications |
| **NetFlow Event** | NetFlow flow records indicate a new host or server | Host/server details derived from exported NetFlow records |

**Exam tip — Quick mapping:**
- Malicious activity affecting availability/integrity/confidentiality → **Intrusion Event**
- Host first appears on the network → **Host or Endpoint Event**
- NetFlow detects a new host → **NetFlow Event** (Network Discovery mechanism)
- Changes detected to known hosts/applications → **Network Discovery Event**
- Sessions between hosts discovered by NGFW → **Connection Event**

## SECTION 5 — SIEM AND SECURITY PLATFORMS

### 5.1 SIEM

**Security Information and Event Management (SIEM)** technology provides real-time reporting and long-term analysis of security events by collecting and correlating log data from across the organization.

SIEM combines **Security Event Management (SEM)** and **Security Information Management (SIM)** into a unified platform.

#### SIEM Core Functions:

| Function | Description |
| :--- | :--- |
| **Log Collection** | Collects event records from sources across the organization for forensic use and compliance |
| **Normalization** | Maps log messages from different systems into a common data model |
| **Correlation** | Links logs and events from disparate systems to speed threat detection |
| **Aggregation** | Reduces event volume by consolidating duplicate event records |
| **Reporting** | Presents correlated data in real-time dashboards and long-term summaries |
| **Compliance** | Generates reports to satisfy regulatory requirements |

#### SIEM Inputs:

- IDS/IPS alerts
- Antimalware devices
- Firewalls
- Full packet captures
- NetFlow telemetry
- Server logs and syslog

#### Popular SIEM Platforms:

| Platform | Notes |
| :--- | :--- |
| **Splunk** | Widely used commercial SIEM in SOCs; made by a Cisco partner |
| **Security Onion with ELK** | Open-source; integrates Elasticsearch, Logstash, and Kibana; includes other NSM tools |

**Exam tip:** The two popular SIEM platforms to know are **Splunk** and **Security Onion with ELK**.

---

### 5.2 SOAR

**Security Orchestration, Automation, and Response (SOAR)** extends SIEM capabilities by **automating security response workflows** and facilitating incident response tasks. It integrates tools from multiple vendors into a unified platform to address the shortage of cybersecurity analysts and reduce response time.

Examples of integrated security platforms: **Cisco SecureX**, **Fortinet Security Fabric**, **Palo Alto Networks Cortex XDR**.

## SECTION 8 — CRITICAL DISTINCTIONS (EXAM TRAPS)

| Concept A | Concept B | Key Difference |
| :--- | :--- | :--- |
| **Session Data** | **Transaction Data** | Session = entire conversation metadata; Transaction = specific request/response within that session |
| **Full Packet Capture** | **NetFlow** | FPC captures the actual payload/content; NetFlow records only metadata (flow records) |
| **Security Logs** | **Application Logs** | Security logs = login/access audits; Application logs = software-specific events |
| **tcpdump** | **nfdump** | tcpdump = FPC tool; nfdump = NetFlow viewing tool |
| **SIEM** | **SOAR** | SIEM = collect, correlate, report; SOAR = automate response workflows |
| **Alert Data** | **Statistical Data** | Alerts = generated by IDS/IPS rules; Statistical = derived from analysis for behavior prediction |
| **NBAR2** | **NetFlow** | NBAR2 = application recognition by signature; NetFlow = flow metadata collection |
| **Failure Audit** | **Error** | Failure Audit = failed security access attempt; Error = system/functional failure (data loss) |
| **Warning** | **Failure Audit** | Warning = low disk space (future problem); Failure Audit = unauthorized access attempt (security) |
